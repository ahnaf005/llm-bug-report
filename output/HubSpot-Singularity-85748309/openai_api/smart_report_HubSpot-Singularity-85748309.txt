Suggested Fixes / Patches
There are two broad correct approaches. Choose one based on desired numeric precision and API design:

1) Change the field types to double (recommended if original source values are double-precision):
- Rationale: avoids narrowing conversions and preserves precision. If upstream Mesos / library returns doubles, representing them as double in your data object is consistent.

Example patch (replace float with double):
- Before:
  private float someMetric1;
  private float someMetric2;
  private float someMetric3;

- After:
  private double someMetric1;
  private double someMetric2;
  private double someMetric3;

- Also update getters/setters/signatures to return/accept double.

2) Keep float fields but perform explicit casts where the narrowing is intentional:
- Rationale: if you deliberately want to store as float to save memory or match a pre-existing API, you must explicitly cast double -> float to satisfy Java's type system.
- Be aware this may lose precision.

Example patch (add casts):
  this.someMetric1 = (float) stats.getSomeMetric1();
  this.someMetric2 = (float) stats.getSomeMetric2();
  this.someMetric3 = (float) stats.getSomeMetric3();

3) Alternative: modify the source of the values so they return float instead of double (less preferred unless you control that API and it's logically float).
