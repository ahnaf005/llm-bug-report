Title
- Build failure caused by Javadoc error in NFAutorizador31.java (bad use of '>')

Summary

  "error: bad use of '>'" in file src/main/java/com/fincatto/nfe310/classes/NFAutorizador31.java.
- The failure is introduced by an unsafe Javadoc comment containing the substring "=>", which the Javadoc tool interprets as a stray '>' (closing angle bracket) and rejects.

Root cause analysis
- The comment line in NFAutorizador31.java contains the literal "=>". The Javadoc tool parses Javadoc comments as HTML (or at least parses tags). A standalone '>' (or sequence that looks like a closing tag without a matching opening '<') triggers a "bad use of '>'" parse error in javadoc (Javadoc treats a raw '>' as stray closing angle bracket).
- The offending comment line is:
    * Links NFC-e segundo site => https://ciranda.me/tsdn/blog-da-tecnospeed/post/como-emitir-nfc-e-na-bahia
  This contains "=>". The '>' character is not escaped or enclosed in {@code} or {@literal}, so javadoc rejects it.
- Many other javadoc warnings are present but are non-fatal; the javadoc parser error is fatal and causes the plugin to exit with code 1.

Code diff (relevant portion)
- The supplied patch shows that the offending comment was changed from:
    * Links NFC-e segundo site => https://...
  to:
    * Links NFC-e segundo site: https://...
- The diff also contains unrelated stylistic changes (reordering an import, adding final to method params, formatting arrays with spaces). Those are not the cause of the javadoc error; the problematic part is the comment operator "=>".

Fixes and recommendations
1. Quick, minimal fix (recommended)
   - Replace the arrow "=>" with a colon ":" or another non-markup character in the Javadoc comment, e.g.:
       * Links NFC-e segundo site: https://...
   - This exactly matches the change in the code diff and will eliminate the javadoc parse error.

2. Robust alternatives (preferred for future-proofing)
   - Escape/encode the '>' using HTML entity:
       * Links NFC-e segundo site =&gt; https://...
     (less preferred than just removing the arrow; still safe)
   - Use Javadoc inline tags to avoid interpretation:
       * Links NFC-e segundo site {@literal =>} https://...
     or
       * Links NFC-e segundo site {@code =>} https://...
   - Wrap the whole line or URL in {@code ...} if you want to preserve special characters:
       * {@code Links NFC-e segundo site => https://...}

3. CI mitigation (temporary)
   - If you need to unblock CI quickly and cannot change source, configure the maven-javadoc-plugin to be non-fatal in CI (not recommended long-term):
     - Use plugin config to set failOnError false for the javadoc plugin (so javadoc errors do not fail build). Example:
         <plugin>
           <groupId>org.apache.maven.plugins</groupId>
           <artifactId>maven-javadoc-plugin</artifactId>
           <version>2.9.1</version>
           <configuration>
             <failOnError>false</failOnError>
           </configuration>
         </plugin>
     - Note: this masks documentation errors and is not recommended as a permanent solution.
