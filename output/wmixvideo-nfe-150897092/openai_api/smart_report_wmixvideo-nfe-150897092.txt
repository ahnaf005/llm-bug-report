Title
- Build failure: Javadoc generation error "bad use of '>'" caused by unescaped '>' in comment in NFAutorizador31.java

Steps to Reproduce
1. Clone repository and checkout the failing commit (the build used commit 12c8a862...).
2. Run Maven install including javadoc generation, e.g.:
   - mvn install -DskipTests=true -Dgpg.skip=true --settings ~/settings.xml
   (or run the maven-javadoc-plugin phase directly: mvn javadoc:jar)
3. Observe build failure during the javadoc generation phase.

Triggering Input / Conditions
- Source file: src/main/java/com/fincatto/nfe310/classes/NFAutorizador31.java
- A Javadoc comment contains the sequence "=>", specifically:
  "* Links NFC-e segundo site => https://ciranda.me/tsdn/blog-da-tecnospeed/post/como-emitir-nfc-e-na-bahia"
- The Maven javadoc plugin (maven-javadoc-plugin:2.9.1) was invoked (via mvn install). Javadoc tool (Standard Doclet) fails parsing the comment.

Expected Behavior
- Maven build completes successfully (javadoc plugin produces apidocs and javadoc jar).
- Javadoc generation should properly handle comments and produce apidocs.

Observed Behavior
- Maven build fails during the javadoc plugin (attach-javadocs) with a Javadoc parsing error:
  - "bad use of '>'" pointing at the comment that contains "=>".

Root Cause Analysis
- The JavaDoc tool treats the '>' character as an HTML tag delimiter or otherwise expects proper markup. The sequence "=>" inside a Javadoc comment can confuse the doclet/parser and produce "bad use of '>'" during javadoc generation.
- The fatal error is triggered during javadoc generation (maven-javadoc-plugin). The compiler step and tests compiled successfully; only the Javadoc phase failed.
- The fix is to avoid unescaped '>' in Javadoc text. The code diff shows a change replacing "=>" with ":" which eliminates the problematic '>'.

Patches / Suggested Fixes
1) Immediate fix (already present in the diff)
- Replace the problematic comment line:
  "* Links NFC-e segundo site => https://ciranda.me/tsdn/blog-da-tecnospeed/post/como-emitir-nfc-e-na-bahia"
  with
  "* Links NFC-e segundo site: https://ciranda.me/tsdn/blog-da-tecnospeed/post/como-emitir-nfc-e-na-bahia"

  This removes the '>' character and fixes the javadoc parse error.

2) Alternative (robust) fixes (recommended)
- Escape the '>' character as an HTML entity:
  "* Links NFC-e segundo site =&gt; https://ciranda.me/..."
- Use {@literal ...} to preserve the literal characters without confusing javadoc:
  "* Links NFC-e segundo site {@literal =>} https://ciranda.me/..."
- Use {@code ...} to render the token as code:
  "* Links NFC-e segundo site {@code =>} https://ciranda.me/..."
- Wrap the entire URL or the whole text in {@literal ...} or {@code ...} where appropriate to make Javadoc treat characters literally.

3) Preventative suggestions
- Run javadoc generation locally or add a CI check to catch Javadoc parsing errors earlier.
- When adding comments that include symbols like <, >, or sequences that look like HTML tags, use {@literal} or escape as &lt; and &gt;.

Summary / Recommendation

- Re-run the build (mvn install -DskipTests=true -Dgpg.skip=true) — the javadoc plugin should then complete and the build should succeed.
